# 生成AIリテラシー学習室 プロジェクト運用

## 目的

1. オーナーが来月の生成AIパスポート試験に合格できる学習環境を作る。
2. 非公式・無料の学習サイトとして公開し、4〜8週間で需要を検証する。
3. 問題・解説・出典・更新履歴を追跡できる、信頼性のある教材運用を確立する。

## 意思決定

- **プロジェクトオーナー（ユーザー）**: 最終承認者。ブランド、公開範囲、計測、外部サービス、有料化、重要な仕様変更を決定する。
- **プロジェクト責任者（Codex）**: バックログの整理、担当割当、成果統合、品質確認、承認依頼、リリース作業を行う。
- **教材責任者**: シラバス対応、問題・解説・出典、難易度と出題配分を管理する。
- **品質責任者**: 自動テスト、セキュリティ、出典追跡、レビュー記録、リリース判定を管理する。
- **プロダクト責任者**: 学習順序、スマートフォン操作、復習、模試、継続利用を管理する。
- **集客責任者**: SEO、計測設計、フィードバック、無料版の需要検証を管理する。

担当は固定の人物ではなく、作業ごとに独立した担当エージェントを割り当てる。プロジェクト責任者は各担当の結論を統合し、相互レビューなしで公開しない。

## 承認ルール

### オーナー承認が必要

- 公開サイトの名称・説明・対象者の変更
- アクセス解析、フォーム、外部サービスの導入
- 個人情報を取得する変更
- 有料化、広告、アフィリエイト
- 大規模な画面構成変更
- 法律・権利分野の新規教材を正式版として公開
- 本番公開への反映

### プロジェクト責任者が進められる

- 調査、監査、修正案、問題・解説の下書き
- テスト、出典確認、セキュリティ修正
- GitHub Issueと作業ブランチの作成
- 公開前のブラウザ検証とレビュー

## 作業フロー

1. GitHub Issueに目的、完了条件、優先度、担当領域を記録する。
2. 担当エージェントが調査または実装する。
3. 別担当が内容・安全性・操作性を確認する。
4. 自動テストとブラウザ確認を実行する。
5. プロジェクト責任者が変更点、リスク、判断事項をオーナーへ提示する。
6. オーナー承認後に`main`へ反映する。
7. GitHub Pages公開後にスモークテストを実行する。

## 優先順位

- **P0**: 誤学習、情報漏えい、重大な操作不能につながる問題。最優先で対応する。
- **P1**: 合格可能性、継続利用、需要検証に大きく影響する問題。
- **P2**: 品質向上、運用効率、将来の成長に必要な改善。

## 現在のリリース判定

公開中のサイトは無料ベータ版として扱う。次の条件を満たすまで正式版とはしない。

- `confidence: "medium"`の公開問題が0件
- すべての問題から根拠資料と確認箇所を追跡できる
- 公式シラバス項目の割当状況を確認できる
- インポート、模試、復習、保存の主要テストが成功する
- P0が0件
- オーナーが正式版公開を承認する

## 運用リズム

- 毎日21時: GitHub Actionsが教材データと一次資料を検査する。
- オーナーが日次作業を起動: チームが最優先Issueを選び、調査・実装・レビューを進める。
- 週1回: 進捗、未解決リスク、次週の優先事項をオーナーへ報告する。
- 4週間後: 学習利用と初期反応を確認する。
- 8週間後: 無料版の継続、方向転換、有料化検討のいずれかをオーナーが判断する。

## 日次指示

オーナーは次の一文だけで作業を開始できる。
